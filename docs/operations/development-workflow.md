# 开发与提交流程

## 目的

本文档定义 `youtube-parser` 的日常开发流程、最小验证要求和文档同步原则。

## 基本原则

### 1. 先契约，再实现

如果改动会影响：

- API 接口
- DTO 字段
- 错误码
- warning 语义

应先更新 `docs/parser-contract.md` 或相关架构文档，再开始实现。

### 2. 小步闭环

一次改动应尽量只表达一个完整逻辑，例如：

- 新增一个 provider
- 调整 transcript 解析
- 修复一种 URL 识别问题
- 补一组契约测试

### 3. 文档与实现同步

当以下内容变化时，应同步更新文档：

- 产品范围
- provider 策略
- 目录结构
- 部署方式
- 契约与错误码

## 标准开发流程

建议采用以下顺序：

1. 阅读 `docs/roadmap/current.md`
2. 阅读 `docs/product/prd.md`
3. 阅读相关架构与模块文档
4. 明确任务边界
5. 实现改动
6. 执行最小验证
7. 更新文档

## 提交前最小验证

### 文档改动

- 文档之间是否一致
- 是否与 `parser-contract.md` 一致

### 代码改动

- 基础语法检查
- 契约测试
- 关键解析样本回归

### 契约改动

- `health / capabilities / parse` 响应是否符合文档
- 错误码是否与文档一致
- 新字段是否为追加式演进

## 何时更新本文档

当以下内容变化时必须更新：

- 开发流程变化
- 验证要求变化
- 文档同步规则变化
